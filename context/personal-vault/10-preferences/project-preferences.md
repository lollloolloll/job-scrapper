# 프로젝트 선호

## 한 줄 요약

사용자는 **실제 사용처, 운영 맥락, 고객/기관 요구사항, 서비스 흐름**이 있는 프로젝트에서 동기와 강점이 잘 살아난다.

## 선호도가 높아 보이는 프로젝트 유형

### 1. 실제 운영 문제를 해결하는 내부 시스템

예시:

- 공간대여 예약 시스템
- 키오스크/대여 관리 시스템
- 관리자 대시보드
- 예약 정책/대여 만료/대기열/통계처럼 운영 조건이 있는 기능

왜 잘 맞는가:

- 사용자가 누군지 명확하다.
- 요구사항이 현실 업무에서 나온다.
- 프론트, DB, 관리자 기능, 정책 로직을 함께 다룰 수 있다.
- 면접에서 “왜 만들었는가”를 설득력 있게 설명할 수 있다.

### 2. 소상공인/상점/고객 유입 중심 웹 서비스

예시:

- 소상상점
- 웹에이전시 공식 홈페이지
- MVP 솔루션 랜딩/상담/문의 흐름
- 소상공인 상점/협업/지도/채팅 UI

왜 잘 맞는가:

- 기능 구현이 매출, 문의, 신뢰, 전환 흐름과 연결된다.
- 단순 기술 프로젝트보다 서비스/비즈니스 관점을 배울 수 있다.
- Product Engineer 지향 포지셔닝과 잘 맞는다.

### 3. 프론트엔드 중심이지만 운영 품질을 챙기는 프로젝트

예시:

- Storybook UI 문서화
- Vercel/Chromatic/next-sitemap/robots
- OAuth/API client/token store
- middleware/static path/SEO 대응

왜 잘 맞는가:

- 단순 화면 구현보다 구조와 품질을 같이 설명할 수 있다.
- 팀 협업과 출시 품질의 근거가 된다.

### 4. AI 도구와 지식관리 workflow를 결합하는 프로젝트

예시:

- Obsidian/Graphify 개인 memory workflow
- AI log index
- curated knowledge map
- Hermes session compact index

왜 잘 맞는가:

- 사용자가 실제로 흥미를 느끼는 영역이다.
- AI를 단순 코드 생성이 아니라 장기 작업 기억과 문서화에 쓰는 차별점이 된다.

## 덜 우선인 프로젝트 유형

- 사용자/운영 맥락 없는 단순 CRUD 토이 프로젝트.
- 공개할 수 없는 고객 프로젝트를 핵심 포트폴리오로 삼는 것.
- 백엔드/인프라 깊이만 요구하는 프로젝트.
- 사업 성과가 확인되지 않았는데 성과 중심으로 과장해야 하는 프로젝트.

## 프로젝트 선택 기준

새 프로젝트를 고를 때 아래 질문을 통과하면 우선순위가 높다.

```text
1. 실제 사용자가 있는가?
2. 운영 문제나 고객 문제를 설명할 수 있는가?
3. 프론트엔드 구현 이상의 흐름/API/DB/운영 정책이 있는가?
4. 이력서나 면접에서 안전하게 공개 가능한가?
5. AI-native workflow나 문서화/자동화와 연결 가능한가?
```

## 추천 포트폴리오 우선순위

1. 소상상점 — 팀 협업형 프론트엔드/서비스 개발 경험.
2. 쌍청문 공간대여 예약 시스템 — 실제 기관 운영 문제를 단독 풀스택으로 해결.
3. 키오스크/대여 관리 시스템 — 키오스크, 관리자, 대여/대기열/만료 처리.
4. 웹에이전시/MVP 프로젝트 — 고객 유입, 랜딩, 상담/문의, 요구사항 정리.
5. AI memory workflow — AI-native 작업 방식과 자기관리 시스템.
